Core claim (what you are testing)

Hypothesis

A runtime policy that permits speculation only when the expected utility is positive can reduce exploitable leakage in high-risk regions while preserving most performance in low-risk regions.

Important: results depend on the evaluation platform, workload selection, and leakage proxy quality.

Decision rule (math)

Rendered using KaTeX (optional).

\(E[U(I_t)] = \pi(t)\cdot G(I_t) - (1-\pi(t))\cdot L(I_t)\)
Continue iff \( \pi(t)G(I_t) > (1-\pi(t))L(I_t) \)

\(\pi(t)\): confidence, \(G\): gain estimate, \(L\): leakage cost (semantic / analysis-derived).

Proposed methodology

Step 1

Define leakage-cost levels

Use a small discrete set (e.g., 2‚Äì4 levels: low/med/high/critical) based on program semantics (secret-dependent loads, transmitter gadgets, etc.).

  • Start coarse (thesis-feasible).
  • Refine later if time permits.
Step 2

Map predictor state ‚Üí confidence bins

Approximate \(\pi(t)\) from predictor metadata (e.g., saturating counters). Calibration can be simple and monotonic.

  • No need for perfect probability.
  • Just consistent confidence ordering.
Step 3

Enforce the inequality cheaply

Avoid division. Use LUT/quantization or log-domain comparison in a small ‚ÄúODU‚Äù unit.

  • Output: speculate / clamp / stall.
  • Goal: minimal critical-path impact.
